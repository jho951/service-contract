# Authz v2 Design

이 문서는 `authz-service`의 v2 확장 방향을 고정한다. v1은 관리자 경로 RBAC 판정에 집중하고, v2는 정책 기반 인가와 권한 수명주기 관리로 확장한다.

## 1. 설계 목표
- 사용자 role, permission, scope를 조회하고 동기화할 수 있게 한다.
- 단순 role 체크가 아니라 정책 기반 인가로 확장한다.
- 권한 변경이 발행하는 세션/토큰 불일치를 다룬다.
- 관리자 권한을 더 세분화하고, 사용자/내부/서비스간 권한을 분리한다.
- 감사 추적과 버전 관리를 계약으로 고정한다.

## 2. Authorization Query APIs
| 항목 | 설명 |
| --- | --- |
| 권한 조회 | 사용자 role, permission, scope, tenant, entitlements를 조회한다. |
| 권한 검증 | 로그인 직후 또는 토큰 검증 시 권한 상태를 동기화한다. |
| introspection | 토큰 유효성뿐 아니라 현재 권한 상태 유효성도 검사한다. |
| UI metadata | 프론트가 메뉴/기능 노출 여부를 판단할 수 있는 메타데이터를 제공할 수 있다. |

### 대표 흐름
1. `GET /v2/permissions/users/{userId}`로 현재 권한 스냅샷 확인
2. `POST /v2/permissions/authorize`로 resource/action/condition 평가
3. `POST /v2/permissions/internal/introspect`로 토큰/권한 상태 재검증

### 예시
```json
{
  "userId": "user-123",
  "resource": "admin:dashboard",
  "action": "read",
  "condition": {
    "tenant": "tenant-a"
  }
}
```
```json
{
  "decision": "ALLOW",
  "authzVersion": "12",
  "policyVersion": "34"
}
```

## 3. Token Claims and Sync
| 항목 | 설명 |
| --- | --- |
| claims | access token에 `roles`, `scopes`, `tenant`, `entitlements`를 반영할 수 있다. |
| policy snapshot | 토큰 발급 시점의 권한 결과를 스냅샷으로 주입할 수 있다. |
| version | `authz_version` 또는 `policy_version`으로 불일치를 감지한다. |
| sync trigger | 로그인, refresh, 권한 변경, 세션 재검증 시 동기화한다. |

### 예시 클레임
```json
{
  "sub": "user-123",
  "roles": ["ADMIN"],
  "scopes": ["admin:read", "admin:write"],
  "tenant": "tenant-a",
  "entitlements": ["feature:reports"],
  "authz_version": 12
}
```

## 4. Policy-based Authorization
| 항목 | 설명 |
| --- | --- |
| model | `resource + action + condition + effect` 형태의 정책 평가를 사용한다. |
| examples | `admin:read`, `project:write`, `user:self:update` 같은 표현을 허용한다. |
| evaluator | role 체크보다 먼저 정책 evaluator를 거치도록 한다. |
| deny by default | 정책이 없으면 거부한다. |
| detailed spec | 세부 문서는 [`policy-model.md`](policy-model.md)를 따른다. |
| runtime | 현재 authz-service는 Maven Central publish된 `plugin-policy-engine`을 evaluator로 사용한다. |

## 5. Cache and Invalidation
| 항목 | 설명 |
| --- | --- |
| authz cache | 권한 서버 호출 비용을 줄이기 위한 TTL 캐시를 둔다. |
| invalidation | role 부여/회수, 그룹 변경, 정책 변경 시 무효화 이벤트를 발행한다. |
| session effect | 이미 발급된 세션/토큰의 재검증 여부를 정책으로 정의한다. |
| gateway coupling | Gateway는 권한 캐시 결과를 보조적으로 사용할 수 있지만 최종 판단 기준은 정책 계약이다. |

## 6. Granular Roles and Delegation
| 항목 | 설명 |
| --- | --- |
| admin split | `ADMIN`을 운영자, 보안관리자, 감사자 등으로 세분화한다. |
| scope split | 사용자용, 내부관리용, 서비스간 호출용 권한을 분리한다. |
| delegation | 조직/팀/프로젝트 단위로 권한 위임을 지원한다. |
| multi-tenant | 테넌트 경계가 있는 경우 tenant-aware policy를 기본 전제로 둔다. |
| detailed spec | 위임 세부 문서는 [`delegation.md`](delegation.md)를 따른다. |

## 7. Audit and Governance
| 항목 | 설명 |
| --- | --- |
| grant/revoke audit | 권한 부여/회수 시점을 추적한다. |
| policy change audit | 정책 변경 이력을 남긴다. |
| actor trace | 누가 어떤 권한을 언제 바꾸었는지 식별할 수 있어야 한다. |
| server scope | authz-service와 auth-service 모두 필요한 범위의 감사 로그를 남긴다. |
| detailed spec | 버전/캐시/경계/실시간 재검증은 [`versioning.md`](versioning.md), [`cache.md`](cache.md), [`external-boundaries.md`](external-boundaries.md), [`introspection.md`](introspection.md)를 따른다. |

## 8. External Responsibilities
| 항목 | 설명 |
| --- | --- |
| auth-service | 로그인/SSO/세션/토큰 발급 원천으로서 권한 스냅샷을 수용한다. |
| gateway | 토큰 검증과 policy application을 함께 수행한다. |
| user-service | 사용자 마스터 데이터를 제공한다. |
| redis | 권한 캐시와 무효화 보조 저장소를 제공한다. |

## 9. Capability and Visibility Boundary
| 항목 | 설명 |
| --- | --- |
| capability truth | `editor:write`, `workspace:write`, `project:edit` 같은 쓰기 권한의 진실은 authz-service가 소유한다. |
| visibility policy | 권한 보유 사실을 프로필에 공개할지 여부는 user-service의 privacy/visibility 정책이 소유한다. |
| final enforcement | 실제 저장/수정/삭제 같은 기능 집행은 Editor 같은 도메인 서비스가 최종 강제한다. |
| separation | 권한의 보유와 공개, 기능 집행을 동일한 책임으로 섞지 않는다. |

## 10. v2 Priority
1. 권한 조회/검증 API
2. 정책 기반 인가
3. 토큰 클레임과 versioning
4. 권한 캐시와 무효화 이벤트
5. 세분화된 관리자 권한
6. 스코프/클라이언트 분리
7. 권한 위임과 멀티테넌시
8. 감사 추적 강화
9. introspection

## 11. Contract Notes
- Authz v2는 최종 인가 기준 서비스다.
- auth-service는 인증과 세션 발급을 담당하고, authz-service는 권한 상태와 정책 판단을 담당한다.
- Gateway는 두 서비스를 조합해 외부 요청을 정규화하고, 관리자 경로는 authz-service 판정을 우선한다.
- 권한의 공개 여부는 user-service 책임이며, capability 진실과 분리해서 문서화한다.
- 정책 evaluator 구현은 `plugin-policy-engine`이 담당하고, 정책 계약은 `policy-model.md`가 담당한다.
