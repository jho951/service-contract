# Editor Error Contract

현재 `editor-service` v1의 에러 코드는 `GlobalResponse<Void>` envelope 안에서 HTTP status와 `code`로 함께 내려간다.
`code` 값은 `ErrorCode` enum과 1:1로 대응한다.

## 응답 형태
```json
{
  "httpStatus": "NOT_FOUND",
  "success": false,
  "message": "요청한 문서를 찾을 수 없습니다.",
  "code": 9004,
  "data": null
}
```

## 코드 정책
- 현재 v1은 `900x`/`901x` 범위를 사용한다.
- `code` 값은 외부 계약이므로 임의 변경하지 않는다.
- HTTP status와 code의 매핑은 `ErrorCode` enum과 일치해야 한다.
- 현재 구현에는 `403`과 `422` 전용 코드는 없다.
- `Node` 단일 영속 모델(v2)의 에러 계약은 별도 문서로 분리한다.

## v1 활성 코드
| HTTP Status | Code | 설명 |
| :---: | :--- | :--- |
| 400 | 9015 | 잘못된 요청입니다. |
| 400 | 9016 | 요청 필드 유효성 검사에 실패했습니다. |
| 405 | 9017 | 허용되지 않은 HTTP 메서드입니다. |
| 401 | 9001 | 인증 정보가 없습니다. |
| 404 | 9002 | 요청하신 URL을 찾을 수 없습니다. |
| 404 | 9004 | 요청한 문서를 찾을 수 없습니다. |
| 404 | 9006 | 요청한 블록을 찾을 수 없습니다. |
| 409 | 9005 | 요청이 현재 리소스 상태와 충돌합니다. |
| 409 | 9007 | 정렬 키 공간이 부족하여 재정렬이 필요합니다. |
| 500 | 9999 | 요청 응답 실패, 관리자에게 문의해주세요. |

## 사용 규칙
- 문서/블록이 실제로 없으면 `404`를 우선한다.
- 인증 정보가 없거나 Gateway 컨텍스트가 누락되면 `401`을 반환한다.
- 버전/정렬 충돌은 `409`로 처리한다.
- 파라미터/입력 스키마 검증 실패는 `400`으로 처리한다.
- 현재 v1에서 `403`과 `422`는 사용하지 않는다.

## 비고
- Gateway 에러 코드와 editor 서비스 에러 코드는 별도 체계다.
- Gateway는 필요 시 editor 서비스의 HTTP status와 `code`를 그대로 전달하거나 표준화할 수 있다.
