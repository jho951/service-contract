# Auth v2 Design

Auth v2는 더 강한 인증과 세션 관리를 위한 목표 계약이다.

## 핵심 원칙
- Gateway가 public `/v2/auth/*` route를 소유한다.
- Auth-service는 인증 동작과 upstream 계약을 소유한다.
- Auth-service upstream route는 기본적으로 `/auth/*`를 유지한다.
- v2 기능 때문에 내부 경로 분리가 꼭 필요할 때만 별도 upstream versioning을 추가한다.
- 최종 권한 판단은 authz-service 계약을 따른다.

## Public Route와 Upstream Route
| Public route example               | Upstream ownership                                                    |
|------------------------------------|-----------------------------------------------------------------------|
| `POST /v2/auth/login`              | Gateway가 route를 받고 Auth-service login behavior로 전달한다.                 |
| `POST /v2/auth/mfa/challenge`      | Gateway public route. Auth-service가 MFA challenge behavior를 제공한다.     |
| `POST /v2/auth/mfa/verify`         | Gateway public route. Auth-service가 MFA verification behavior를 제공한다.  |
| `GET /v2/auth/sso/start/{provider}` | Gateway public route. Auth-service가 provider 시작 behavior를 제공한다.       |
| `GET /v2/auth/sessions`            | Gateway public route. Auth-service가 session inventory behavior를 제공한다. |
| `POST /v2/auth/logout-all`         | Gateway public route. Auth-service가 session revocation behavior를 제공한다. |

## v2 목표 범위
| 영역 | 목표 |
| --- | --- |
| MFA / step-up | TOTP, WebAuthn, recovery code, admin 강제 MFA |
| SSO provider | GitHub 외 Google, Apple, Kakao, SAML, OIDC 확장 |
| Session lifecycle | session inventory, revoke-by-device, logout-all |
| Token introspection | token 유효성뿐 아니라 session/account 상태까지 확인 |
| Account policy | active, locked, suspended 상태 전이와 password lifecycle |
| Internal management | password reset, lock/unlock, MFA 관리, social unlink |
| Audit / risk | MFA, risk event, device, session, request correlation |

## MFA / Step-up
### 목표
- 1차 password 또는 SSO 인증 이후 추가 인증이 필요한 상태를 표현한다.
- MFA 완료 전에는 관리자 또는 고위험 권한을 가진 최종 session/token을 발급하지 않는다.
- `MfaFactor`는 사용자별 등록 factor의 저장 모델이다.

### Public Flow
1. Client가 `POST /v2/auth/login`을 호출한다.
2. Gateway가 Auth-service login behavior로 전달한다.
3. 추가 인증이 필요하면 응답에 `mfaRequired=true`와 challenge 생성에 필요한 값을 내려준다.
4. Client가 `POST /v2/auth/mfa/challenge`를 호출한다.
5. Client가 `POST /v2/auth/mfa/verify`로 code 또는 credential을 검증한다.
6. 검증 성공 시 최종 access token / refresh token / session을 발급한다.

### Response Example
```json
{
  "accessToken": null,
  "refreshToken": null,
  "mfaRequired": true,
  "challengeId": "mfa-challenge-123"
}
```

## SSO Provider Expansion
현재 v1 구현 기준 provider는 GitHub다. v2 목표는 provider별 구현을 registry 형태로 분리하는 것이다.

Required behavior:
- provider별 authorize URL 생성
- provider별 callback 처리
- 공통 ticket exchange
- provider별 client id, secret, issuer, redirect URI 설정
- provider 추가 시 controller 분기보다 provider registry 확장을 우선

Example:
```txt
GET /v2/auth/sso/start/google?page=editor&redirect_uri=https://app.example.com/auth/callback
```

## Session / Token Lifecycle
v2는 “로그인 여부”만 확인하지 않고 “어떤 기기/세션이 살아 있는지”를 관리한다.

Required behavior:
- 현재 사용자의 session list 조회
- 특정 session revoke
- 전체 session revoke
- device id 또는 client fingerprint 연결
- 동시 session 제한
- high-risk action 전 introspection

Example response:
```json
{
  "sessions": [
    {
      "sessionId": "session-123",
      "deviceId": "device-abc",
      "createdAt": "2026-04-01T09:30:00Z",
      "lastUsedAt": "2026-04-14T09:30:00Z"
    }
  ]
}
```

## Account Policy
v2 account policy는 단순 lock flag를 넘어 상태 전이를 명시한다.

States:
- `active`: 정상 사용 가능
- `locked`: 실패 횟수, 위험 신호, 관리자 조치로 잠김
- `suspended`: 운영 정책상 사용 중지

Required behavior:
- failed login count 기반 lock
- password updated/expired timestamp 관리
- 관리자 lock/unlock
- 위험 기반 step-up 또는 re-auth
- status change audit event 발행

## Internal Management
v2 internal API는 User-service 또는 운영 도구가 Auth-service 계정 상태를 관리하기 위한 API다.

Planned actions:
- password reset
- account lock/unlock
- MFA factor 등록/폐기
- MFA 강제 재등록
- social unlink
- account status sync

## Audit / Risk Signals
Audit event는 사용자의 인증 상태가 바뀌는 순간을 추적해야 한다.

Events:
- password login success/failure
- SSO login success/failure
- MFA challenge created
- MFA verified/failed
- refresh
- logout
- session revoke
- account lock/unlock
- high-risk re-auth required

Risk signals:
- IP
- user-agent
- device id
- geo
- failed attempts
- session id
- request id / correlation id

## Implementation Priority
1. MFA / step-up response model
2. MFA challenge and verify behavior
3. session inventory and revoke
4. token/session introspection
5. provider registry
6. account lock/unlock and status sync
7. audit/risk event enrichment

## Contract Notes
- v2는 public contract의 기능 확장이다. public route prefix는 Gateway가 소유한다.
- Auth-service 구현은 public API version prefix에 결합하지 않는다.
- Auth-service가 내부 경로 versioning을 도입할 경우 이 문서가 아니라 별도 upstream API 계약에서 명시한다.
