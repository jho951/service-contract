# User Visibility & Privacy Contract

`User-server`는 사용자 프로필의 공개 범위, 개인정보 가시성, 권한 노출 정책을 소유한다.
`Authz-server`는 어떤 권한/스코프를 보유하는지의 진실을 소유하고, `Editor` 같은 기능 서버는 최종 실행 시점을 강제한다.

## 책임
| 책임 | 설명 |
| --- | --- |
| 프로필 가시성 | 사용자 프로필 필드가 다른 사용자, 관리자, 내부 시스템에 어떻게 보이는지 결정한다. |
| 공개 범위 | 특정 필드나 배지, 상태를 공개/관리자 전용/비공개로 분리한다. |
| 권한 노출 정책 | `editor:write`, `workspace:write`, `project:edit` 같은 capability를 프로필에 노출할지 결정한다. |
| 개인정보 정책 | 권한 보유 사실을 다른 사용자에게 보여줄지, 관리자에게만 보여줄지, 완전히 숨길지 결정한다. |

## 책임 분리
| 영역 | 책임 주체 | 설명 |
| --- | --- | --- |
| 쓰기 권한의 진실 | `Authz-server` | 사용자가 실제로 `editor:write`, `workspace:write`, `project:edit` 같은 권한을 보유하는지 판정한다. |
| 공개 여부 | `User-server` | 그 권한/역할을 프로필이나 사용자 공개 정보로 노출할지 결정한다. |
| 최종 기능 집행 | `Editor` | 실제 문서/블록/워크스페이스 수정 시점에 권한을 최종 강제한다. |

## 예시
| 상황 | Authz 판단 | User 공개 정책 | Editor 실행 결과 |
| --- | --- | --- | --- |
| 내부 관리자 전용 배지 | `editor:write = ALLOW` | 관리자만 표시 | 기능 수행 가능 여부와 별개로 표시 여부는 제한 |
| 일반 사용자 공개 프로필 | `workspace:write = ALLOW` | 비공개 | 다른 사용자에게는 권한 보유 사실을 숨김 |
| 감사자 전용 열람 | `project:edit = ALLOW` | 감사자/관리자에게만 표시 | 실제 편집은 editor가 별도로 검증 |

## 계약 원칙
- 권한의 진실은 `Authz-server`가 소유한다.
- 권한의 공개 여부는 `User-server`가 소유한다.
- 기능 사용 가능 여부는 `Editor`가 최종적으로 강제한다.
- 공개 프로필에 보이는 권한 표시는 실제 쓰기 권한을 대체하지 않는다.
- 프로필 공개 정책은 `user-service`의 privacy/visibility 정책으로 문서화하고, 새로운 API가 생기면 `CONTRACT_SYNC.md`를 함께 갱신한다.

## v2 확장 포인트
- 사용자 프로필 응답에 공개 범위 메타데이터를 추가할 수 있다.
- 관리자용 프로필 조회와 일반 사용자용 프로필 조회의 노출 필드를 분리할 수 있다.
- authz 결과를 그대로 노출하지 않고, 별도의 표시 정책으로 정규화할 수 있다.
