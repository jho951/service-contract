# Auth v2 Design

이 문서는 `Auth-server`의 v2 확장 방향을 고정한다. v1은 로그인, refresh, logout, GitHub SSO, 내부 계정 생성에 집중하고, v2는 인증 강도와 세션/계정 수명주기를 확장한다.

## 1. 설계 목표
- MFA와 step-up 인증을 인증 흐름에 자연스럽게 통합한다.
- SSO provider를 플러그인 구조로 확장한다.
- 세션과 토큰의 수명주기 관리를 강화한다.
- 계정 정책과 내부 관리 API를 운영 친화적으로 확장한다.
- v1/v2 라우트, DTO, 쿠키 정책, 응답 포맷을 분리한다.

## 2. MFA / Step-up Authentication
| 항목 | 설명 |
| --- | --- |
| 대상 | TOTP, WebAuthn, 복구 코드, 관리자 계정 강제 MFA |
| 저장 모델 | `MfaFactor` 엔티티를 인증 플로우와 연결한다. |
| 트리거 | 일반 로그인, 고위험 작업, 관리자 계정, 새 기기, 위험 기반 정책 |
| 단계 분리 | 1차 인증 성공 후 step-up 챌린지를 별도 상태로 관리한다. |
| 보안 원칙 | MFA 완료 전에는 관리자/고위험 세션을 발급하지 않는다. |

### 대표 흐름
1. `POST /v2/auth/login`
2. 응답에서 `mfaRequired=true`면 `POST /v2/auth/mfa/challenge`
3. 사용자가 TOTP/WebAuthn/복구 코드로 `POST /v2/auth/mfa/verify`
4. 검증 성공 시 최종 `accessToken` / `refreshToken` 발급

### 예시
```json
{
  "username": "user@example.com",
  "password": "password1234"
}
```
```json
{
  "mfaRequired": true
}
```
```json
{
  "challengeId": "mfa-challenge-123",
  "code": "123456"
}
```
```json
{
  "accessToken": "eyJhbGciOi...",
  "refreshToken": "eyJhbGciOi...",
  "mfaRequired": false
}
```

## 3. SSO Provider Expansion
| 항목 | 설명 |
| --- | --- |
| provider 모델 | GitHub, Google, Apple, Kakao, SAML, OIDC를 provider 플러그인으로 분리한다. |
| 진입점 | `startOAuthLogin`과 SSO controller는 provider registry를 통해 라우팅한다. |
| callback | provider별 callback 처리와 공통 ticket exchange를 분리한다. |
| 설정 | provider별 client/issuer/redirect 설정을 독립적으로 관리한다. |
| 정책 | provider 추가는 코드 수정이 아니라 계약/설정 확장으로 처리한다. |

### 대표 흐름
1. `GET /v2/auth/sso/start/{provider}`
2. provider redirect 후 callback
3. `POST /auth/exchange` 또는 v2 상응 엔드포인트로 ticket 교환

### 예시
```txt
GET /v2/auth/sso/start/google?page=editor&redirect_uri=https://app.example.com/auth/callback
```

## 4. Session / Token Lifecycle
| 항목 | 설명 |
| --- | --- |
| session inventory | 기기별 세션 목록을 조회할 수 있어야 한다. |
| logout-all | 모든 세션을 한 번에 종료하는 흐름을 제공한다. |
| revoke-by-device | 특정 디바이스 세션만 취소할 수 있어야 한다. |
| introspection | 토큰 유효성뿐 아니라 현재 세션/상태 유효성도 확인한다. |
| device binding | 토큰에 디바이스 바인딩 정보를 선택적으로 반영한다. |
| 동시 로그인 제한 | 정책에 따라 계정 단위 동시 세션 수를 제한한다. |

### 대표 흐름
1. `GET /v2/auth/sessions`로 현재 세션 목록 확인
2. 특정 세션은 `DELETE /v2/auth/sessions/{sessionId}`로 종료
3. 전체 종료는 `POST /v2/auth/logout-all`
4. `POST /v2/auth/internal/session/introspect`로 고위험 요청 전에 상태 재검증

## 5. Account Policy
| 항목 | 설명 |
| --- | --- |
| 상태 전이 | `active`, `locked`, `suspended` 상태를 명시적으로 관리한다. |
| 비밀번호 만료 | 비밀번호 만료 정책과 재설정 토큰을 지원한다. |
| 잠금 해제 | 관리자 또는 정책 기반 잠금 해제 흐름을 제공한다. |
| 위험 기반 잠금 | IP, device, failed attempts, geo 이상징후를 반영한다. |
| 마지막 상태 | 마지막 로그인 시간, 비밀번호 수정 시간, 잠금 원인을 함께 유지한다. |

## 6. Internal Management APIs
| 항목 | 설명 |
| --- | --- |
| account lifecycle | 계정 생성/삭제에 더해 비밀번호 초기화, 잠금/해제, 상태 동기화를 제공한다. |
| MFA management | MFA 등록, 폐기, 강제 재등록을 제공한다. |
| social unlink | 소셜 연동 해제를 내부 관리 API로 제공한다. |
| sync API | 외부 시스템과의 계정 상태 동기화를 위한 조회 API를 제공한다. |

## 7. Versioning and Contracts
| 항목 | 설명 |
| --- | --- |
| route split | `/auth/*`와 `/v1/auth/*`를 명시적으로 분리한다. |
| DTO split | v1과 v2의 request/response DTO를 분리한다. |
| cookie policy | 브라우저 쿠키 정책은 버전별로 명확히 구분한다. |
| response policy | error code, response envelope, redirect contract를 분리한다. |
| compatibility | v1은 유지하되 v2는 기능 확장용 경로로 운영한다. |

## 8. Audit and Risk Signals
| 항목 | 설명 |
| --- | --- |
| audit scope | 성공/실패/SSO/logout/internal account뿐 아니라 MFA, risk event, 권한 변경을 포함한다. |
| telemetry | IP, user-agent, device id, geo, anomalous login 신호를 수집한다. |
| correlation | 감사 로그는 session, device, request, user를 연결할 수 있어야 한다. |
| risk action | 고위험 이벤트는 step-up 또는 재인증을 유도한다. |

## 9. v2 Priority
1. MFA / step-up 인증
2. 세션 수명주기와 introspection
3. SSO provider 확장
4. 계정 정책 고도화
5. 내부 관리 API 확장
6. 감사/리스크 신호 확장
7. 라우트/DTO 버전 분리

## 10. Contract Notes
- Auth v2는 인증 원천으로서의 책임을 유지하되, 권한 판단은 Authz-server와 분리한다.
- v2에서 필요한 권한 스냅샷은 토큰 발급 시점에 주입할 수 있지만, 최종 인가는 Authz-server의 계약을 따른다.
